# Wave 0 — Dual Pilot (Subsections 1.1 + 2.2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce two pilot research reports — `research/01_economic_machine.md` (subsection 1.1, Economic Machine Template) and `research/09_all_weather.md` (subsection 2.2, All-Weather Portfolio) — that each pass every item in the acceptance criteria checklist. Compare them and capture lessons in a single retrospective that proposes specific prompt refinements for Wave 1.

**Why two pilots:** Per advisor review. Subsection 1.1 is foundational/narrative — hardest test of Hard Rule R4 (≤15% theory). Subsection 2.2 is naturally numeric — cleanest test. Running both in parallel lets us distinguish "prompt weakness" from "subsection resisting the schema."

**Architecture:** Stand up shared research infrastructure (deterministic prompt template with embedded subsection registry, acceptance-criteria checklist) so all 12 subsections reuse the same artifacts. Spawn **two** pilot subagents in parallel, one per subsection. Evaluate both outputs against the same checklist. Write a unified retrospective. Present to user. Iterate the template before fanning out Waves 1–3.

**Tech stack:**
- Markdown for prompt template, criteria, retrospective, reports
- Markdown table for subsection registry (inline in the prompt template file)
- `Agent` tool with `subagent_type: "general-purpose"` for research subagents
- Git + GitHub (repo live at https://github.com/Siriko404/RayDalioAttitude)
- Bash + Edit/Write for all file operations

**Non-goals for Wave 0:**
- Do NOT spawn more than two agents
- Do NOT start Wave 1 until user approves the refined prompt template
- Do NOT build any dashboard/Excel artifact yet
- Do NOT mechanize the prompt-filling step (manual substitution is fine for 2 agents)

### Task 2: Create acceptance criteria checklist

**Files:**
- Create: `research/_acceptance_criteria.md`

- [ ] **Step 1: Write the file**

Every line is an objective pass/fail test. No subjective "good quality" — only "section exists," "table has N columns," "word count between X and Y." Use this structure:

```markdown
# Acceptance Criteria — Research Subagent Reports

Every agent report (`research/<SEQ>_<slug>.md`) MUST pass ALL items below.
A single FAIL = report rejected and re-run (with refined prompt if needed).

## Structural criteria (format / presence)

- [ ] S1. File exists at `research/<SEQ>_<slug>.md` matching the registry.
- [ ] S2. Word count is between 2000 and 3000 (use `wc -w`).
- [ ] S3. Exactly 10 top-level sections titled `## § 1` through `## § 10`, in order.
- [ ] S4. Each section has the EXACT title prescribed by the prompt template's REQUIRED OUTPUT SCHEMA.
- [ ] S5. § 1 Executive Summary is ≤ 100 words.
- [ ] S6. § 4 Input Variables Table has columns: name | description | unit | data source | API endpoint | update frequency | typical range.
- [ ] S7. § 8 has three sub-sections: 8a JS, 8b Excel, 8c ECharts — all present.

## Content criteria (hard rules R1-R9)

- [ ] R1. Every required section (10) is present — satisfied by S3/S4.
- [ ] R2. Every numeric threshold / formula has a source citation (Dalio or non-Dalio marker).
- [ ] R3. Every input variable in § 4 has a specific public data source + API endpoint / dataset ID.
- [ ] R4. Word-count estimate: ≥85% of the text is inputs / formulas / computation / worked example / implementation; ≤15% is narrative theory. (Eyeball + word-count of § 2+§ 3 vs § 4-§ 8.)
- [ ] R5. Any ambiguity in Dalio's writing is explicitly flagged in § 10, not papered over.
- [ ] R6. No content belongs to another subsection per the registry's OUT-OF-SCOPE list.
- [ ] R7. Every claim has a visual attribution marker: either `> **Dalio** — source: ...` or `> **NON-DALIO (industry standard)** — source: ...`. Check:
       ```bash
       grep -cE '^> \*\*(Dalio|NON-DALIO)' research/<SEQ>_<slug>.md
       ```
       Expect ≥ 8 markers for a 2000-3000 word report. If 0, attribution was silently skipped — FAIL.
- [ ] R8. Every cited source has a public URL. No paywalled, library-only, or login-gated links. Check:
       ```bash
       grep -oE 'https?://[^ )]+' research/<SEQ>_<slug>.md | sort -u
       ```
       Manually scan the list — flag any paywalled domain (jstor, sciencedirect, wiley, tandfonline etc. WITHOUT a matching NBER/SSRN/Levy free version linked).
- [ ] R9. No commercial-book quote exceeds 1 sentence; cumulative quotes from any one commercial book ≤ 2 sentences.

## Palette-compliance criteria (§ 8c)

- [ ] P1. Every hex color in § 8c is one of the locked palette tokens from spec § 13. Check:
       ```bash
       grep -oE '#[0-9a-fA-F]{6}' research/<SEQ>_<slug>.md | sort -u
       ```
       Expected allowed set (exactly): `#0B0B0B`, `#141414`, `#1C1C1C`, `#080808`, `#262626`, `#F5F5F5`, `#A3A3A3`, `#6B7280`, `#00D08C`, `#7FFFD4`, `#E5484D`, `#D4A373`. Any hex NOT in this list = FAIL.

## Closeout

- [ ] C1. § 9 lists at least one upstream and one downstream integration point (even if "none").
- [ ] C2. § 10 lists all sources with full citations + public URLs.
```

- [ ] **Step 2: Verify the file**

Run: `grep -c "^- \[ \]" "research/_acceptance_criteria.md"` → expect **exactly 19** checkboxes (S1–S7 + R1–R9 + P1 + C1–C2).
Manually scan: no item says "should be good" / "appropriate" / "sufficient" — every item is objective.

- [ ] **Step 2: Write the retrospective file**

Structure:

```markdown
# Wave 0 Retrospective

**Date:** <today>
**Pilots:** 01_economic_machine (subsection 1.1), 09_all_weather (subsection 2.2)

## 1. Checklist results

| Item | Pilot A (1.1) | Pilot B (2.2) | Notes |
|---|---|---|---|
| S1 | PASS | PASS | |
| S2 | PASS (2417 words) | PASS (2650 words) | |
| ... one row per checklist item (19 rows) ... |

## 2. Pattern findings (what both pilots got right / wrong)

- What worked in BOTH: <observations>
- What failed in BOTH: <observations — these are prompt problems, not subsection problems>
- Where A and B diverged: <observations — these flag subsection-specific issues>

## 2.5 VERDICT (pick exactly one)

This is the whole point of the dual pilot — force a verdict that distinguishes prompt failure from subsection resistance.

- [ ] **VERDICT-P (prompt failure):** both pilots fail one or more of the SAME checklist items → the prompt template is the problem → refine the prompt and re-run Wave 0 before Wave 1.
- [ ] **VERDICT-S (subsection resistance):** only 1.1 fails items that 2.2 passes (or vice versa) → the subsection resists the schema, not the prompt → accept as a known limitation and tighten that subsection's SCOPE_IN / SCOPE_OUT individually. Wave 1 can launch.
- [ ] **VERDICT-B (both):** shared failures AND subsection-specific failures → partial prompt refinement PLUS per-subsection scope tightening needed.

Supporting evidence: list the exact checklist items driving the verdict.

## 3. Proposed prompt refinements for Wave 1

Each refinement is specific and testable (a diff, not a wish):

1. **Refinement #1:** <exact text to add / change in `_prompt_template.md`> — addresses <which failed checklist items> — ties to <VERDICT-P or VERDICT-B>.
2. **Refinement #2:** ...
3. **Refinement #3:** ...

## 4. Recommendation

- [ ] Launch Wave 1 with refinements applied (if VERDICT-S, or VERDICT-B with minor prompt changes)
- [ ] Re-run one pilot with refinements before Wave 1 (if VERDICT-P with significant changes)
- [ ] Abort Wave 0 and escalate (if prompt needs fundamental rethink)
```
